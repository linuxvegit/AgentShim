# Plan 03 — Gemini Native Adapter (AI Studio)

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Source design:** [`docs/superpowers/specs/2026-04-30-phase-2-provider-breadth-design.md`](../specs/2026-04-30-phase-2-provider-breadth-design.md) (decisions D5, D3, D11).

**Goal:** Land the Gemini native adapter targeting `generativelanguage.googleapis.com` (AI Studio) with API-key auth. This is the largest plan in Phase 2 because Gemini's wire format is genuinely different from the OpenAI-Chat shape — it uses `Content`/`parts`, JSON-array streaming (not SSE), and a thinking-budget mechanism that needs custom translation.

**Architecture:**
- New sibling provider module `crates/providers/src/gemini/` that does **not** compose `oai_chat_wire` — different wire format. Has its own request encoder and response parser.
- The exception is `interleaved_reasoning.rs` from `oai_chat_wire/`: Gemini's `parts` carry a `thoughts: bool` flag indicating reasoning vs content, and the same state machine drives the interleaved-block emission. Plan 02 ships the state machine; Plan 03 imports it.
- A new `streaming_json.rs` module implements an incremental JSON-array parser that emits one element at a time. Required because Gemini streams as a JSON array `[{...}, {...}, {...}]`, not SSE. This is the highest-risk piece of new code in v0.2.
- AI-Studio API-key auth: `?key=AIza...` URL query parameter. No OAuth, no service accounts. Vertex variants deferred to v0.2.x.

**Tech stack:** No new dependencies. The JSON-array parser is built on `bytes` + `serde_json::Deserializer::from_slice` invoked incrementally. Tests reuse `mockito`.

**Core changes:** NONE (frozen-core, ADR-0002). Gemini safety ratings land in `extensions["gemini.safety_ratings"]` on `CanonicalResponse`.

### Task 3: AI-Studio auth + endpoint

- [ ] `auth.rs`: a small struct `AiStudioAuth { api_key: SecretString }` with one method `apply(builder: RequestBuilder) -> RequestBuilder` that adds `?key={api_key}` as a query parameter.
- [ ] `endpoint.rs`: URL builders for the two endpoints v0.2 uses:
  - `streaming_url(base_url, model)` → `{base_url}/models/{model}:streamGenerateContent?alt=sse` ⚠️ **CHECK** whether AI Studio honors `alt=sse` for our purposes; if it returns SSE on this path, it's a different code path than the JSON-array streaming. **Default assumption per D5**: streaming uses JSON-array format; `alt=sse` is **not** used. The streaming URL is `{base_url}/models/{model}:streamGenerateContent`.
  - `unary_url(base_url, model)` → `{base_url}/models/{model}:generateContent`
- [ ] **Implementation note:** Some Gemini docs reference `?alt=sse` returning SSE. If during implementation you find `alt=sse` works and produces SSE, switch to that — it's strictly easier than JSON-array. Document the choice in `docs/providers/gemini.md`. Otherwise stick with JSON-array per Q5's risk acceptance.

### Task 4: Request encoder

- [ ] `request.rs`: `pub fn build(req: &CanonicalRequest, target: &BackendTarget) -> GenerateContentRequest`:
  - Map canonical roles: `User` → `"user"`, `Assistant` → `"model"`, `Tool` → `"function"` (Gemini-specific role for tool responses).
  - Map `ContentBlock::Text` → `Part { text: Some(...), .. }`.
  - Map `ContentBlock::Image` → `Part { inline_data: Some(InlineData { mime_type, data }), .. }` (base64 string from `BinarySource::Base64` or `Bytes`; URL forms become `Part { file_data: ... }` if Gemini's `file_data` shape is supported, otherwise the encoder fetches and embeds as base64 — **decide and document** during implementation; bias toward "URL passes through as `file_data` when possible").
  - Map `ContentBlock::ToolCall` → `Part { function_call: Some(...), .. }`.
  - Map `ContentBlock::ToolResult` → `Part { function_response: Some(...), .. }`.
  - Map `ContentBlock::Reasoning` → `Part { text: Some(...), thought: Some(true) }` (the symmetric of how the response decodes thinking).
  - System instructions: aggregate `req.system` into a single `Content { role: "user", parts: [{text}] }` for the `system_instruction` field.
  - `tools` from canonical → Gemini `Tool { function_declarations: [...] }`.
  - `generation_config.thinking_config.thinking_budget` from `req.resolved_policy.reasoning.budget_tokens` (Anthropic-style) or a fixed mapping from `req.resolved_policy.reasoning.effort` (`minimal=128, low=256, medium=1024, high=4096, xhigh=16384` — pick reasonable defaults; document in `docs/providers/gemini.md`).
- [ ] Unit tests covering each mapping case.

### Task 5: JSON-array streaming parser

- [ ] `streaming_json.rs`: incremental parser. State machine:

```
state ::= ExpectOpenBracket | InElement { depth: u32 } | BetweenElements | Done

byte stream → buffer accumulator.
On every chunk:
  while there's progress:
    case ExpectOpenBracket: skip whitespace; expect '['; transition to BetweenElements.
    case BetweenElements:   skip whitespace; if ']' transition Done. If '{' start InElement.
                            If ',' stay BetweenElements.
    case InElement:         track brace depth + string state. When depth returns to 0,
                            slice that range out as a serde_json::Value, emit element,
                            transition BetweenElements.
    case Done:              consume trailing whitespace; warn on extra bytes.
```

- [ ] Returns a `Stream<Item = Result<serde_json::Value, ParseError>>`.
- [ ] **String-state tracking is non-negotiable**: brace depth alone can't distinguish `}` inside a string literal from a real closing brace. Track `in_string`, `escape_next`, `in_object_depth`, `in_array_depth` separately.
- [ ] Unit tests: well-formed array with N elements (N=0,1,many), elements containing strings with `}` and `]`, escapes (`\"`, `\\`), Unicode code points, malformed (early EOF, missing comma), elements split across chunk boundaries (every 1, 2, 3, …, 1024 bytes).
- [ ] Fuzz test: `tests/json_array_streaming_fuzz.rs` — feed arbitrarily-chunked input bytes from a known-good fixture, assert parser emits the same elements as parsing the whole buffer at once.

### Task 6: Response parser

- [ ] `response.rs`: streaming path consumes the JSON-array parser's output, decoding each element as a `GenerateContentResponse`. Per element:
  - Walk `candidates[0].content.parts`. Each `part.thought == true` → push (`Reasoning`, text) into the `ReasoningInterleaver` (D3). Else push (`Text`, text).
  - On `function_call` part → emit `ContentBlockStart { ToolUse }` + `ToolCallStart` + `ToolCallArgumentsDelta` for the JSON-stringified args.
  - On `inline_data` part → response-side image; v0.2 logs at debug level and drops, per D6 risk acceptance.
  - On `finish_reason` (`STOP`, `MAX_TOKENS`, `SAFETY`, `RECITATION`, `OTHER`) → flush interleaver, emit `MessageStop` with mapped `StopReason` (`SAFETY → ContentFilter`, `RECITATION → Refusal`, others map cleanly).
  - On `safety_ratings` → store on `extensions["gemini.safety_ratings"]` of the `CanonicalResponse` (build the response container in the parser; emit `ResponseStop` with the populated extensions at the end).
- [ ] Unary path: same logic without streaming — fetch full body, map once.
